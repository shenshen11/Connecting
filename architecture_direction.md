# 类 PICO VR 互联原型：架构方向说明

## 1. 文档目的

本文档用于明确我们接下来要开发的“类 PICO VR 互联”闭环原型的技术方向，避免项目一开始就陷入过度产品化、过度抽象或纯 Demo 化。

我们的目标不是立即复刻官方 PICO 互联产品，而是先做出一个 **可验证、可演进、可逐步原生化** 的系统原型。

---

## 2. 本项目目标

### 2.1 第一阶段目标：闭环 Demo

实现如下完整链路：

1. PC 端运行 Unity 场景
2. PICO 4 Ultra 头显端采集头部姿态（HMD Pose）
3. 头显端将姿态实时发送到 PC
4. PC 根据最新姿态更新 Unity 中的远端相机
5. PC 渲染左右眼画面
6. PC 将画面编码并发送到头显
7. 头显解码并显示双目画面
8. 用户转头时，看到画面同步变化

这条链路跑通后，项目就从“想法”变成了“真实系统原型”。

---

## 3. 总体架构结论

结合当前讨论，我们不采用“纯 Unity 双端”的方案，而采用以下 **混合架构**：

- **PC 端：Unity + Windows Native 模块**
- **头显端：Android Native + OpenXR + MediaCodec**
- **控制/信令：一个轻量服务或最小控制通道**
- **共享协议层：统一定义 pose、视频、控制消息结构**

该方案的核心思想是：

- **Unity 负责内容生产与场景渲染**
- **原生模块负责实时链路与底层性能关键路径**

这是一个兼顾开发速度与后续可演进性的技术路线。

---

## 4. 为什么不做纯 Unity 双端

纯 Unity 双端的优点是开发快，但存在明显问题：

1. 视频采集、编码、解码链路不够可控
2. 头显显示路径受 Unity XR 封装限制较大
3. 后续做低延迟优化时容易受限
4. 很多真正关键的问题（纹理接入、编码器参数、MediaCodec、OpenXR 显示层）会被框架包住

因此，纯 Unity 更适合“快速演示一个概念”，但不适合作为我们后续继续深入的主干架构。

---

## 5. 最终采用的混合架构

### 5.1 PC 端

PC 端运行在 Windows 上，使用 Unity 作为场景与渲染容器，但关键链路由原生模块实现。

#### PC 端分工

**Unity 负责：**

- 场景加载与逻辑
- 左右眼相机管理
- 接收并应用远端 HMD 姿态
- 渲染左右眼图像
- 调试 UI 与开发期可视化

**Windows Native 模块负责：**

- 从 Unity 获取渲染纹理 / GPU 纹理
- 执行左右眼纹理拼接或编码前处理
- 调用硬件编码器（优先 NVENC）
- 发送编码视频流
- 接收头显上传的 pose 数据
- 提供时间戳与统计信息

#### 预期实现形式

- Unity Windows App
- C++ DLL / Unity Native Plugin
- 图形后端优先考虑 D3D11

---

### 5.2 头显端（PICO 4 Ultra）

头显端不再以 Unity 为主，而是采用 Android Native 应用。

#### 头显端职责

- 使用 OpenXR Native 获取 HMD pose
- 按高频率发送 pose 数据给 PC
- 接收 PC 发来的编码视频流
- 使用 MediaCodec 硬件解码视频
- 将解码后的视频显示到 XR 视图中
- 管理连接、重置朝向、基础状态显示

#### 头显端技术方向

- Android Studio 工程
- Java / Kotlin 负责 Activity 壳层与系统集成
- C++（NDK）负责实时逻辑
- OpenXR Native 负责姿态与显示时序
- MediaCodec 负责硬件解码
- OpenGL ES 或 Vulkan 负责图像提交

---

### 5.3 控制/信令层

第一阶段不做复杂后台，只需要最小控制能力。

职责包括：

- 建立 PC 与头显的连接关系
- 交换初始化参数（IP、端口、编解码配置等）
- 支持后续扩展重连、状态查询、日志拉取

第一阶段可简化为：

- 直接配置 IP + 固定端口
- 或极小的本地/局域网信令服务

重点不是后台，而是闭环链路本身。

---

## 6. 系统数据流

### 6.1 上行：头显 -> PC

头显实时发送：

- 头部位置（Position，可选先弱化）
- 头部旋转（Rotation，第一阶段重点）
- 时间戳
- 序列号
- 可能的状态位（追踪状态、重定位状态等）

### 6.2 下行：PC -> 头显

PC 发送：

- 双目视频流（第一阶段可用 SBS：Side-by-Side）
- 视频时间戳
- 基础控制消息

### 6.3 本地闭环

1. 头显采样 pose
2. pose 发到 PC
3. PC 取最新 pose 更新相机
4. Unity 渲染双目图像
5. Native 模块编码视频
6. 视频发送到头显
7. 头显解码并显示

---

## 7. 第一阶段范围控制

为了保证项目推进，我们明确限制第一阶段范围。

### 7.1 第一阶段必须完成

- 头显 pose 原生采集
- pose 上行到 PC
- Unity 场景根据 pose 更新视角
- PC 原生视频编码
- 头显原生视频解码
- 双目显示
- 闭环运行

### 7.2 第一阶段不做

- 控制器输入
- 音频链路
- SteamVR / OpenXR Runtime 对外兼容
- 自适应码率
- 复杂丢包恢复
- 产品级 UI
- 账号 / 配网 / 后台系统
- 官方级低延迟重投影优化

这一步的目标只是：**证明系统闭环成立**。

---

## 8. 推荐技术选型

### 8.1 Windows 端

- Unity：用于场景与相机
- 原生插件：C++ DLL
- 图形接口：D3D11（优先）
- 编码器：NVIDIA NVENC（优先目标）
- 网络：UDP（第一阶段可自定义简单协议）

### 8.2 Android / PICO 端

- Android Studio
- Java / Kotlin：Activity、权限、基础生命周期
- C++ / NDK：实时逻辑主实现
- OpenXR Native：pose / frame loop / display
- MediaCodec：硬件解码
- OpenGL ES：第一阶段优先，便于落地

### 8.3 共享协议层

定义统一的数据结构，包括：

- PosePacket
- VideoPacketHeader
- ControlPacket
- Timestamp / Sequence 规范
- 坐标系与重置规则

## 10. 开发顺序

这是最关键的一节。我们必须严格按顺序推进，不能一开始就并行做全部问题。

### 阶段 0：架构和协议定稿

先完成：

- pose 数据结构定义
- 视频包头定义
- 坐标系约定
- 时间戳规范
- Unity / Native 边界划分

输出物：

- 协议草案
- 线程模型说明
- 基础目录结构

---

### 阶段 1：头显端原生 pose 上行

目标：

- Android Native + OpenXR 工程跑起来
- 读取 HMD pose
- 用 UDP 发到 PC

验收标准：

- PC 能收到连续的姿态数据
- 头显转头时，PC 端日志中的 rotation 持续更新

这是第一个必须打通的链路。

---

### 阶段 2：PC 端 Unity 消费 pose

目标：

- Unity 接收 pose 数据
- 在场景中驱动一个相机或调试物体

验收标准：

- 不需要视频链路
- 仅通过 PC 屏幕就能看到 Unity 场景视角随头显姿态变化

这是第二个里程碑，表示“姿态驱动远端渲染”成立。

---

### 阶段 3：PC 端原生编码链路

目标：

- Unity 输出左右眼纹理
- Native 插件获取纹理
- 编码为 H.264 / HEVC
- 本地或局域网发送视频流

验收标准：

- 能输出连续编码视频
- 有基本帧率与编码耗时统计

---

### 阶段 4：头显端原生解码显示

目标：

- Android 接收视频流
- MediaCodec 解码
- 显示到 XR 视图中

验收标准：

- 头显能看到来自 PC 的远端画面
- 暂不要求最终体验最优

---

### 阶段 5：整链路合并，形成闭环

目标：

- 头显 pose -> PC
- PC 更新 Unity 相机
- PC 渲染与编码
- 头显接收与解码显示

验收标准：

- 用户头动时，头显中看到的画面同步变化
- 能稳定运行数分钟

到这一步，闭环 Demo 完成。

---

### 阶段 6：最小优化与可演示化

包括：

- 重置朝向（Recenter）
- 只消费最新 pose，避免排队
- 输出基础统计
- 简单连接配置
- 日志与故障定位能力

---

## 11. 当前最重要的工程原则

### 原则 1：先打通 pose 上行，再做视频
因为姿态链路是系统的核心。

### 原则 2：先做 rotation，position 可后置精调
第一阶段最关键的是头转动的感知闭环。

### 原则 3：先做最短链路，不做过早产品化
不要一开始引入控制器、音频、SteamVR 等复杂度。

### 原则 4：关键性能路径尽量放在原生层
编码、解码、纹理桥接、显示应尽量可控。

### 原则 5：所有优化必须基于统计与时间戳
避免盲目“凭感觉优化”。

---

## 12. 后续演进方向（第二阶段以后）

在闭环 Demo 成立之后，可以继续往以下方向扩展：

1. 控制器输入上行
2. 音频下行 / 麦克风回传
3. 更完善的时间同步
4. 预测与低延迟重投影
5. 网络抖动与丢包补偿
6. 更直接的显示层优化
7. 与 OpenXR / SteamVR 生态进一步对接

这些都不是当前第一阶段的阻塞项。

---

## 13. 当前结论

本项目的第一阶段，不应定义为“做一个完整产品”，而应定义为：

> 构建一个基于 Unity 场景、Windows 原生编码链路、Android Native 头显端、可完成头显姿态回传和远端双目显示闭环的系统原型。

这个原型一旦成立，后续就具备明确的演进基础。

---

## 14. 下一步建议

建议马上进入以下动作：

1. 建立目录结构
2. 明确 PC / Android / Shared 三个工程边界
3. 先起 Android Native + OpenXR 的 pose sender
4. 同时准备 Unity 端 pose receiver 与调试相机

也就是说，接下来第一条开发主线应当是：

> **先做头显原生 pose 上行链路。**
