# 第一阶段原生开发计划（Unity 后置版）

## 1. 目标

第一阶段不直接接入 Unity，而是优先完成最关键、最不确定的原生链路验证：

1. 在 PICO 4 Ultra 上通过 Android Native + OpenXR 采集 HMD 姿态
2. 通过 UDP 将姿态发送到 Windows
3. 在 Windows 原生程序中接收、解析并打印或可视化姿态
4. 搭好后续视频编码、视频解码、Unity 接入所需的目录结构与协议基础

这一步的目的不是做完整产品，而是先把最底层、最容易卡住的能力跑通。

---

## 2. 本阶段交付物

- `shared-protocol/`
  共享协议定义，包括包头、pose 结构、基础控制消息结构等

- `windows-native/`
  Windows 原生姿态接收端与后续发送端工程骨架

- `android-native/`
  Android Native + OpenXR 头显端工程骨架

- 中文说明文档
  用于指导后续阶段如何接着推进

---

## 3. 任务顺序

### Step 1：定义共享协议

先统一好这些内容：

- 包头格式
- pose 数据结构
- 时间戳语义
- 坐标系约定
- 序列号与版本号规则

### Step 2：实现 Windows Pose Receiver

先做一个最小 UDP 接收器：

- 监听固定端口
- 校验包头
- 解析 pose 包
- 打印位置、旋转、时间戳、序列号

### Step 3：建立 Android Native 工程骨架

先把头显端 Native 工程搭起来，包含：

- NativeActivity 或等价原生入口
- OpenXR 初始化骨架
- pose sender 占位
- UDP 发送器

### Step 4：完成第一轮真机联调

在用户本地环境具备以下条件后进入联调：

- Android SDK / NDK
- OpenXR loader
- PICO 设备调试环境
- ADB 连接正常

---

## 4. 本阶段不做什么

第一阶段明确不做以下内容：

- Unity 场景接入
- 视频编码 / 解码完整实现
- 控制器输入
- 音频链路
- 重投影
- 自适应码率
- 产品级 UI

原因很简单：这些内容会显著拉高复杂度，但并不影响我们先验证原生闭环的可行性。

---

## 5. 第二阶段计划

当 pose sender 和 receiver 跑通之后，下一阶段按下面顺序推进：

1. Windows 端测试图像生成与编码发送
2. Android 端接收视频并用 MediaCodec 解码显示
3. 打通原生视频闭环
4. 最后把 Unity 作为内容源接进来

---

## 6. 当前阶段的工程原则

- 先做 pose，再做视频
- 先做原生，再接 Unity
- 先做可验证闭环，再做产品体验
- 关键性能路径必须掌握在原生层

---

## 7. 本阶段完成的标志

当满足以下条件时，可以认为第一阶段成功：

- PICO 4 Ultra 上的原生 OpenXR 应用稳定运行
- Windows 端能够持续接收并解析来自头显的 pose
- 姿态数据频率、稳定性和字段格式满足后续联调用途

达到这一步后，项目就从“架构设想”进入“真实系统实现”阶段。
